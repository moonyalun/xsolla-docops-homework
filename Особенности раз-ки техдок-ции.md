## Как определить комплект необходимой документации

### Определите тип документации

Определите тип документации: внешняя или внутренняя.
Для этого задайте вопросы:
-   Какая бизнес-цель?
-   Кто заказчик?

Тип документа, бизнес-цель и заказчик – это некоторые из [атрибутов](#атрибуты) любого документа.

### Подготовьте список документов с требованиями
Составить список нормативных документов с требованиями к документации:
- ГОСТ
- Документы с требованиями от регуляторов
- Документы с требованиями к документации СМК организации

### Определите цель, целевую аудиторию и вид каждого документа комплекта

### Характеристики технического документа
Любой технический документ можно анализировать с точки зрения его характеристик:
-   атрибутов
-   свойств
-   сущностей

#### Атрибуты документа

| Атрибут           | Ответ на вопрос                                                                        |
|-------------------|----------------------------------------------------------------------------------------|
| Цель              | Какую проблему пользователя решает документ?                                           |
| Бизнес-цель       | Какую проблему проекта/продукта решает документ?                                       |
| Заказчик          | Кому нужен этот документ? Кто заказывает разработку документа (заинтересованное лицо)? |
| Целевая аудитория | Кто будет читать документ? Кому предназначен?                                          |
| Представление     | В какой форме будет представлен документ? Формат конечного документа (doc, pdf, excel, html)                  |

#### Свойства документа



| Свойство                 | Описание                           | Примеры                                             |
|--------------------------|------------------------------------|-----------------------------------------------------|
| Тип                      | Внешний/внутренний                 |                                                     |
| Вид                      | Общее представление документа      | руководство, глоссарий, wiki                        |
| Объем                    | Размеры, объемные характеристики   | Количество листов, приложений, носителей информации |
| Широта целевой аудитории | Уровень квалификации пользователей | Новичок, продвинутый                                 |
| Структура                | глубина (best practices – не больше 4-5) и сечение (best practices – не больше 9)                 |
|                          | длина предложений/абзацев          | Короткие, емкие |
|                          | наличие унифицирующих фигур описания (фраз) | выберите пункт меню, выполните одно из следующих действий, нажмите кнопку/клавишу, приведено в таблице|
|                          | наличие маркированных структурных элементов | примечания, внимание, подсказка, выделение списков |
|                          | наличие иллюстративных элементов   | скриншоты, иконки, видеотуториалы                    |

|                 | наличие логических повторов и перекрестных ссылок  | есть, например, справки по другим продуктам, отсылки к разделам с др. информацией                                                    |



#### Сущности документа
Концепция сущностей предложена компанией [Divio](https://documentation.divio.com/) в 2017 году на PyCon Australia.

-   Пояснения/описания
-   Tutorial (руководство/инструкция/getting started/quick start)
-   How-to
-   Cправочники

Сущности могут представлять собой самостоятельные документы или комбинироваться в составе одного документа.


| Tutorial  | How-to                             | Справочник                               | Пояснение\описание             |
|-----------------------------------------------|------------------------------------|------------------------------------------|----------------------------------|
| Ориентирован на обучение                     | Проблемно-ориентированное описание | Имеет информационное значение            | Служит для объяснения, понимания |
| В форме урока                                 | Состоит из шагов, как рецепт       | Сухое изложение фактов, как энциклопедия | Имеет форму истории, рассказа    |

### Требования к документации

Из [списка документов](#норм_требования) определите для выделенных [свойств](#свойства) документа:
#### Требования к содержанию
-   стиль изложения соответствует виду документа и целевой аудитории
-   соблюдается единство обращения к пользователю документации
-   соблюдается единство терминологии
-   соблюдаются орфография и стилистика языка
-   для описания однотипных действий используются шаблонные структуры
#### Требования к оформлению
-   использование различных шрифтов (жирный, курсивный, моноширинный)
-   использование списков (нумерованный. маркированный)
-   перекрестные ссылки

#### Определите терминологию
-   сущности компании
-   сущности предметной области
-   аббревиатуры
-   форматы данных
-   отдельные фразы

### Принципы документирования
При работе с текстом документа анализируйте на применение следующих принципов документирования

| Принцип                      | Описание принципа                                        | Как добиться                  |
|------------------------------|----------------------------------------------------------|--------------------------------------|
| Целесообразность             | Контент соответствует цели документа                     | <ul><li>Знать [назначение документа](#цель_ца_вид).</li><li> Тестировать документацию. </li><li> Ставить себя на место пользователя, учитывать [проклятие знания](#проклятие_знания).</li></ul>                                 |
| Актуальность                 | Документация соответствует последнему состоянию продукта | <ul><li>Учитывать изменения в документации при подготовке релизов/новых версий программы. </li><li>Периодически проводить ревизию документации.</li></ul>   |
| Фактологическая точность     | Документация не содержит фактических ошибок              | <ul><li>Тестировать документацию.</li><li>Обращаться к достоверным источникам, [нормативным документам](#норм_требования).</li></ul>                                   |
| Структурированность          |<li>Прослеживается логическая структура.</li><li>Используются маркированные и нумерованные списки.</li><li>Повествование ведется от общего к частному (пирамида Минто)</li><li>Термины определяются до первого использования.</li>|<ul><li> Тестировать документацию. </li><li>Ставить себя на место пользователя, учитывать [проклятие знания](#проклятие_знания).</li><li>Разрабатывать шаблоны документов.</li></ul>                                |
| Однозначность                | Контент понятен пользователю после одного прочтения      | <li>Не перегружать предложения сложными языковыми конструкциями.</li><li>Использовать простые и завершенные предложения</li><li> Тестировать документацию. </li><li>Ставить себя на место пользователя, учитывать [проклятие знания](#проклятие_знания).</li> |
| Полнота                      | Исчерпывающий контент для достижения цели                | <li> Тестировать документацию. </li><li>Ставить себя на место пользователя, учитывать [проклятие знания](#проклятие_знания).</li>                                   |
| Консистентность терминологии | <li>Единая терминология.</li><li>Единый стиль изложения.</li>                                    | <li> Использовать глоссарий. </li><li>Ставить себя на место пользователя, учитывать [проклятие знания](#проклятие_знания).</li>                        |

## Общие проблемы при разработке документа
-   Текст не адаптирован под целевую аудиторию
-   Документ не структурирован
-   Нет объяснения сложных терминов и аббревиатур
-   Не учитывается обратная связь от читателя/пользователя
-   Отсутствие визуализации

### Проклятие знания
Проклятие знания – термин для обозначения психологического феномена, при котором более информированным людям сложнее рассматривать какую-либо проблему с точки зрения менее информированных людей.

Рекомендации:
-   при вычитке собственных документов представляйте, уровень знаний пользователя документации, что вы знаете/не знаете о предметной области
-   рассматривайте более низкий уровень квалификации пользователя, будет ли этот документ понятен новичку?

### Принцип пирамиды Минто

-   Структурируйте мысли в форме пирамиды
-   Группируйте похожие идеи
-   Используйте обоснования методом дедукции (главная идея, второстепенная идея, заключение)
-   Используйте обоснования методом индукции (общий вывод на основе схожих суждений и объединяющего их признака)
-   Подходите к проблеме с систематической точки зрения и визуализируйте ее с помощью логических деревьев/диаграмм/схем
-   Используйте введение, чтобы в первые 30 секунд чтения рассказать читателю о ключевых моментах документа
-   Используйте заголовки и форматирование, чтобы продемонстрировать структуру пирамиды

## Best practies

-   Изучайте предметную область
-   Ставьте себя на место пользователя
-   Шаблонизируйте структуру документов
-   Делайте относительно небольшие глубину вложенности (не больше 4) и сечение (не больше 9)
-   Делайте формулировки простыми, лаконичными, завершенными и однозначными
-   Используйте настоящее время вместо будущего, или обращайте внимание на учет времени при разработке документа, например при описании реализованных и проектируемых функций
-   Используйте действительный залог вместо страдательного
-   Не используйте причастные, деепричастные обороты отглагольные существительные
-   Не используйте уточнения в скобках без необходимости
-   Соблюдайте баланс между заботой о пользователе и рамками стиля изложения
-   Вовремя останавливайтесь
-   Делайте перерывы
-   Используйте рецензентов
